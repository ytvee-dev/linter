# Обзор правил

Этот документ описывает группы правил, которые получает пользователь `@ytvee/linter`. Это не полный внутренний каталог каждого правила ESLint или SonarJS, а практический обзор поведения публичных профилей.

## Общая модель

Профили собираются из нескольких слоев:

- игнорируемые пути;
- глобальные переменные браузера и Node.js;
- рекомендуемые правила ESLint;
- правила JavaScript и TypeScript;
- правила импортов;
- отключение конфликтов с Prettier;
- React-слой для JSX и хуков;
- SonarJS-слой для дополнительных диагностик;
- строгий слой для более жестких TypeScript-ограничений.

## Игнорируемые пути

ESLint-профили пакета пропускают служебные и собранные файлы:

```text
node_modules
dist
.eslintrc*
.prettier*
tsup*
examples
.yarn
.git*
*.md
*.json
*.yml
Dockerfile
.pnp.*
```

Prettier запускается отдельно и не использует этот список ESLint-игнорирования.

## Базовые правила

Базовый слой используется всеми публичными профилями.

Он включает:

- рекомендуемые правила `@eslint/js`;
- глобальные переменные браузера и Node.js;
- запрет опасных конструкций вроде `eval`, `new Function`, расширения встроенных объектов и `__proto__`;
- предпочтение `const` вместо `let`, если переменная не переназначается;
- запрет `var` в JavaScript-файлах;
- строгие сравнения через `eqeqeq` с разрешением сравнения с `null`;
- запрет лишних `return`, лишних тернарных выражений и бесполезной конкатенации строк;
- предупреждение для вложенных тернарных выражений;
- сокращенный синтаксис объектов;
- шаблонные строки вместо конкатенации;
- ограничение `Symbol` и `BigInt` через `no-restricted-syntax`.

## TypeScript

Для `.ts` и `.tsx` включен набор `typescript-eslint` с проверкой типов. Профили используют `projectService`, поэтому TypeScript должен видеть корректный `tsconfig.json`.

Дополнительно пакет задает правила:

| Правило                                            | Поведение                                                                           |
| -------------------------------------------------- | ----------------------------------------------------------------------------------- |
| `@typescript-eslint/no-explicit-any`               | В обычных профилях выключено, в строгих профилях включено.                          |
| `@typescript-eslint/explicit-function-return-type` | Требует явный тип возврата функций.                                                 |
| `@typescript-eslint/consistent-type-definitions`   | Для объектных типов предпочитает `interface`.                                       |
| `@typescript-eslint/no-unused-vars`                | Запрещает неиспользуемые значения, но разрешает имена с `_`.                        |
| `@typescript-eslint/explicit-member-accessibility` | Требует `public`, `protected` или `private` для членов класса, кроме конструкторов. |
| `@typescript-eslint/member-ordering`               | Проверяет порядок полей, конструкторов и методов класса.                            |

Пример разрешенного неиспользуемого параметра:

```ts
function handleClick(_event: Event): void {
  save();
}
```

## Импорты

Пакет включает `eslint-plugin-import` и `eslint-plugin-simple-import-sort`.

Проверяется:

- запрет дублей импортов через `no-duplicate-imports`;
- запрет расширений для импортов JavaScript и TypeScript-файлов;
- сортировка импортов по группам;
- автоисправление порядка импортов при запуске `ytvee-linter fix`.

Группы сортировки:

1. внешние пакеты;
2. пакеты с областью имени;
3. абсолютные алиасы;
4. импорты с побочными эффектами;
5. относительные импорты из родительских каталогов;
6. относительные импорты из текущего каталога.

## React, JSX и доступность

React-слой применяется к `.jsx` и `.tsx`.

Он включает:

| Правило                                           | Поведение                                                   |
| ------------------------------------------------- | ----------------------------------------------------------- |
| `react/jsx-filename-extension`                    | JSX разрешен только в `.jsx` и `.tsx`.                      |
| `react/jsx-boolean-value`                         | Для `true` булевое свойство пишется без значения.           |
| `react/self-closing-comp`                         | Компоненты без детей должны быть самозакрывающимися.        |
| `react/jsx-key`                                   | Элементы в списках должны иметь `key`.                      |
| `react/jsx-no-duplicate-props`                    | Запрещает дубли свойств JSX.                                |
| `react-hooks/rules-of-hooks`                      | Проверяет правила вызова хуков.                             |
| `react-hooks/exhaustive-deps`                     | Предупреждает о неполных зависимостях хуков.                |
| `jsx-a11y/alt-text`                               | Требует альтернативный текст для изображений.               |
| `jsx-a11y/anchor-is-valid`                        | Проверяет корректность ссылок.                              |
| `jsx-a11y/click-events-have-key-events`           | Требует клавиатурную обработку для кликабельных элементов.  |
| `jsx-a11y/no-autofocus`                           | Предупреждает об `autoFocus`.                               |
| `jsx-a11y/no-noninteractive-element-interactions` | Предупреждает о событиях на неинтерактивных элементах.      |
| `jsx-a11y/no-static-element-interactions`         | Предупреждает о событиях на статических элементах без роли. |

## SonarJS

SonarJS включается в профилях:

- `@ytvee/linter`;
- `@ytvee/linter/configs/default`;
- `@ytvee/linter/configs/sonar`;
- `@ytvee/linter/configs/react-sonar`.

Общие правила SonarJS применяются к `.js`, `.mjs`, `.cjs`, `.jsx`, `.ts` и `.tsx`. Правила, которым нужны сведения о типах, применяются только к `.ts` и `.tsx`.

Команда `ytvee-linter fix` без локального `eslint.config.*` SonarJS не включает.

## Строгие профили

Строгий слой включается в:

- `@ytvee/linter/configs/strict`;
- `@ytvee/linter/configs/strict-react`.

Он добавляет:

- запрет `any`;
- соглашение именования переменных в `camelCase` или `PascalCase`;
- разрешение начального и конечного `_`;
- ограничение одного крупного объявления на файл: класс, интерфейс, тип или перечисление.

## Prettier

Prettier не запускается как ESLint-правило. Его запускают команды:

```bash
ytvee-linter format
ytvee-linter format --check
ytvee-linter fix
```

Настройки:

```json
{
  "printWidth": 120,
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "all"
}
```
